# App Layer

This folder is reserved for the eventual app shell once platform choice is explicitly confirmed. Keep it thin: compose `src/domain`, `src/workflow`, and `src/modules` here, but do not push UI or platform concerns back down into those layers.

Current implementation seam:

- `src/app/br03TouchpointSnapshotProducer.ts` converts persisted `OUT-04` touchpoint posture source events into BR03 `touchpointPostureSnapshots` with explicit registry-default fallback for missing source events.
- `src/app/outputHandoffCheckpointHydration.ts` wires persisted `OUT-04` source events through the snapshot producer into output/handoff composition checkpoint hydration, and now also supports BR04 lifecycle orchestration checkpoint resume loading/replay with controlled no-record handling and fail-loud malformed payload handling.
- `src/app/launcherCurrentMatterLifecycleResumeAdapter.ts` maps BR04 `outputPackageLifecycleResume` checkpoint status into neutral launcher/current-matter resume routing metadata (available vs no-record, hold/release visibility, and deletion-vs-deidentification route visibility) without adding UI wording or compliance claims.
- `src/app/launcherCurrentMatterExecutionRouting.ts` provides the narrow launcher/current-matter execution-routing seam that consumes neutral lifecycle resume routing metadata and resolves contract-faithful outcomes (`RESUME_AVAILABLE`, `NO_LIFECYCLE_RECORD_FOUND`, `CANNOT_SAFELY_RESUME_RECORD`) without adding UI wording or compliance claims.
- `src/app/launcherCurrentMatterExecutionCaller.ts` adds the first internal launcher/current-matter caller for lifecycle execution routing by consuming output-checkpoint execution-routing resolution and returning neutral internal control outcomes while preserving raw routing metadata for downstream inspection.
- `src/app/launcherCurrentMatterFollowOnRouteCoordinator.ts` wires launcher/current-matter lifecycle internal control outcomes into explicit follow-on internal route-handling outcomes, including fail-safe cannot-resume and explicit no-routing-signal handling without adding UI wording or compliance claims.
- `src/app/launcherCurrentMatterTransitionSelector.ts` threads launcher/current-matter follow-on coordinator decisions into internal transition targets and handling modes only, preserving no-record, fail-safe, and explicit no-signal protections without defining UI/copy/status/CTA behavior.
- `src/app/launcherCurrentMatterTransitionOrchestrationEntry.ts` composes lifecycle transition selection with non-lifecycle transition inputs at a narrow orchestration entrypoint while keeping lifecycle and non-lifecycle transition state separately inspectable and non-certifying.
- `src/app/launcherCurrentMatterExecutionDirectiveConsumer.ts` consumes launcher/current-matter transition orchestration decisions and maps them into internal execution directives only (`DIRECTIVE_*`), while preserving lifecycle/non-lifecycle inspectability and explicit no-record, fail-safe, and no-signal handling without introducing UI/status/CTA/screen-route semantics.
- `src/app/launcherCurrentMatterHandlingActionConsumer.ts` consumes launcher/current-matter execution directives and maps them into internal handling actions only (`ACTION_*`), while preserving lifecycle/non-lifecycle inspectability and explicit no-record, fail-safe, and no-signal handling without introducing UI/status/CTA/screen-route/public-route semantics.
- `src/app/launcherCurrentMatterLifecycleConsumptionPlanner.ts` implements the narrow `SCRC-01` internal lifecycle consumption planner seam from `P4C-CX-APP-CONSUME-02`, consuming existing internal route/directive/handling outputs to produce internal-only non-visual lifecycle planning outcomes while preserving protected-state distinctions and explicit non-rendered/non-CTA/non-analytics authorization flags.
- `src/app/launcherCurrentMatterWatchpointLoggingEmitter.ts` emits internal-only guardrail/watchpoint events from launcher/current-matter handling outcomes using a schema-frozen, minimised payload contract (`WATCH_*` families), preserving no-record, fail-safe, no-signal, hold/release, and deletion/de-identification distinctions without introducing UI/status/CTA/public analytics semantics.
- `src/app/launcherCurrentMatterWatchpointWiredCaller.ts` threads explicit optional watchpoint sink injection through one narrow launcher/current-matter output-checkpoint caller path so handling outputs can emit internal `WATCH_*` observability events when requested, while remaining inert when no sink is provided.
- `src/app/launcherCurrentMatterWatchpointRegressionDiagnostics.ts` adds a bounded internal `WLB-04` regression-diagnostics harness around the existing one-path sink-injected watchpoint caller, using explicit injected in-memory test sinks only to detect protected-state drift and forbidden event/payload semantics without expanding runtime logging adoption topology.
- `src/app/outputPackageLifecycleOrchestration.ts` wires one concrete BR04 output-package lifecycle consumer path at adapter level: output-package record construction, class-based lifecycle runtime planning, scoped hold/release command handling, deletion-vs-deidentification route selection, RBAC checks, audit emission capture, and replayable orchestration records.
- `src/app/outputPackageLifecycleOrchestrationPersistence.ts` adds a thin persistence seam for `OutputPackageLifecycleOrchestrationRecord`: stable-key save/load, malformed/missing fail-loud behavior, and replay-through-existing-planner support without introducing provider-specific storage claims.
