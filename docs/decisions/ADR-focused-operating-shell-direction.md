# ADR: Focused Operating Shell Direction

Date: 2026-04-22
Task ID: P4C-REPO-DOC-02
Status: accepted

## Context

The repo was still easy to read as a narrow arrears-only skeleton, while Product, Legal, and Marketing alignment now supports a bounded operating shell above the hero arrears workflow.

Contributors needed a single decision record to avoid:

- drifting toward generic PM-suite framing
- overclaim language around reminders or exports
- accidental phase confusion between 4B and 4C

## Decision

Landlord Buddy adopts a focused operating shell above the hero arrears-to-notice-readiness workflow, with these controls:

- hero wedge remains first and lead path
- shell capabilities remain within the accepted inclusion list in `docs/specs/focused-operating-shell-baseline.md`
- shell exclusions remain explicit and enforced
- prep-and-handoff doctrine remains unchanged
- no legal-advice behavior
- no official-system implication
- no reminder implies compliance certification
- no export implies filing, acceptance, or legal sufficiency
- phase posture remains 4B primary with 4C parallel product-experience construction

## Consequences

- Repo docs should describe a bounded shell plus hero wedge, not a wizard-only product and not a generic PM suite.
- Architecture, QA, and specs indexes should expose the shell baseline, posture, and semantic checklist.
- Any proposed shell scope expansion or implication-sensitive wording change is a review event requiring explicit decision follow-up.

## Revisit Triggers

- approved change to shell inclusion/exclusion set
- approved change to anti-overclaim or reminder/export implication rules
- approved phase transition beyond current 4B primary / 4C parallel posture
